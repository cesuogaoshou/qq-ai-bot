# 开发规约

## 总原则

项目优先保证可控、稳定、低风险。任何功能都应服务于以下目标：

- 不刷屏。
- 可关闭。
- 可限频。
- 可定位问题。
- 不泄露隐私。
- 不把协议端细节扩散到业务代码。

## 范围控制

MVP 只支持一个固定群。实现时可以保留多群扩展字段，但不要提前实现复杂多群管理。

禁止在 MVP 中加入：

- Web 管理后台。
- 插件市场。
- 图片理解。
- 语音识别。
- 自动踢人、禁言。
- 长期复杂用户画像。
- 本地大模型部署。
- Docker 强依赖。

## 模块边界

必须保持以下边界：

- `onebot` 只处理协议连接、事件解析和消息发送。
- `router` 只做事件分发。
- `policy` 只判断是否回复。
- `llm` 只处理提示词和模型调用。
- `memory` 只处理存储。
- `services` 编排业务流程。

不要让 OneBot 事件处理函数直接调用大模型，也不要让大模型客户端直接发送群消息。

## 配置约定

- 非敏感配置放在 YAML 文件。
- API Key、access token 放在环境变量。
- 配置文件中只保存环境变量名称，不保存真实密钥。
- 单群 MVP 使用 `target_group_id` 白名单。
- 后续多群扩展时再改为 `allowed_group_ids` 或数据库群配置。

## 日志约定

必须记录：

- 启动和配置加载结果。
- OneBot 连接成功、断开、重连。
- 收到目标群消息。
- 触发原因，例如 `mention`、`command`。
- 不回复原因，例如 `disabled`、`rate_limited`、`not_mentioned`。
- 大模型调用耗时、超时和失败。
- OneBot 发送成功或失败。
- 管理员命令执行结果。

日志不应记录：

- API Key。
- access token。
- 过长的完整上下文。
- 不必要的隐私字段。

## 安全约定

- 默认回复模式为 `mention_only`。
- 回复长度必须有限制。
- 大模型调用必须设置超时。
- 命中限频时不调用大模型。
- 管理员关闭后不应响应普通 @。
- 争吵、攻击、隐私、违法或高风险指导场景下应拒绝、降温或不回复。
- 不推断用户敏感属性。

## 存储约定

SQLite 存储第一版只保存必要数据：

- 群配置。
- 最近群消息。
- 用户 ID 和昵称。
- 限频状态。
- 必要的运行状态。

最近消息应有数量上限。MVP 建议保留最近 30 条上下文用于模型调用，数据库中可按时间或数量定期清理。

## 测试约定

MVP 至少覆盖：

- 普通消息不触发回复。
- @ 消息触发回复。
- 非目标群消息被忽略。
- `/bot on` 生效。
- `/bot off` 生效。
- `/bot status` 返回状态。
- 群限频生效。
- 用户限频生效。
- 大模型超时时流程不崩溃。
- OneBot 发送失败时记录日志。

优先测试 `policy`、命令解析、限频和 prompt 构造，因为这些模块最容易影响群聊体验。

## 错误处理

- OneBot 断开后应记录日志并尝试重连。
- 大模型超时后应放弃本次回复或发送短错误提示，不能阻塞后续消息。
- 发送失败只记录错误，不重复刷屏重试。
- 配置缺失时启动失败并给出明确错误。
- SQLite 写入失败时记录错误，不能导致无限循环。

## 代码风格

后续实施时建议：

- 使用类型标注。
- 使用 `async` 编排 I/O。
- 业务对象使用清晰的数据模型。
- 避免把大段业务逻辑写在入口文件。
- 单个模块职责明确，文件过大时拆分。
- prompt 文本集中管理，避免散落在业务代码中。

## 变更原则

新增功能必须回答三个问题：

1. 是否会增加刷屏或风控风险？
2. 是否有管理员开关和限频？
3. 是否能通过日志定位问题？

如果答案不清楚，不进入 MVP。
