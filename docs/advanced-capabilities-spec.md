# 高级能力规格

## 目标

本规格承接 M2 之后的新需求：

- 联网搜索有时效性的资料。
- 阅读和总结群聊记录。
- 基于聊天记录形成有限、可控的群友特征记忆。
- 安全处理敏感话题、污言秽语、争吵和个人隐私，但不过度拒答。
- 支持输出图片和读图。
- 评估语音转文字能力。
- 所有能力必须考虑模型能力、调用价格、群聊体验和 QQ 协议风险。

当前默认文本模型保持为火山方舟豆包：

```text
LLM_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
LLM_MODEL=doubao-seed-2.0-lite
```

OpenAI 兼容接口只是客户端协议形态，不代表默认供应商切换为 OpenAI。

## 总体原则

高级能力不应直接塞进主回复链路，而应作为可控工具：

```text
群消息
  -> 触发策略
  -> 安全策略
  -> 成本策略
  -> 工具选择
  -> 模型回复
  -> 输出过滤
```

默认策略：

- 普通聊天不联网搜索。
- 普通聊天不生成图片。
- 普通聊天不处理语音。
- @ 机器人时才考虑调用大模型。
- 搜索、图片、语音必须有单独开关、限频和成本预算。
- 管理员必须能关闭高级能力。

## 联网搜索

### 使用场景

以下问题可以触发搜索：

- 明确包含“最新”“今天”“最近”“现在”“刚刚”“价格”“政策”“版本”“新闻”等时效词。
- 用户明确要求“帮我查”“搜一下”“联网查”。
- 模型回答需要当前资料，否则容易过时。

以下问题不触发搜索：

- 普通闲聊。
- 群内梗和气氛回应。
- 不需要实时资料的知识问答。
- 涉及隐私、攻击、违法或高风险指导的问题。

### 输出要求

搜索回答必须：

- 简短说明“我查到的信息是……”。
- 给出 1-3 条来源摘要。
- 对不确定信息明确说明不确定。
- 不把搜索结果原文大段复制到群里。

### 成本控制

搜索工具必须支持：

- 每群每日搜索次数上限。
- 每用户每日搜索次数上限。
- 单次搜索结果条数上限。
- 管理员开关。

第一版建议：

```text
默认关闭自动搜索。
只允许显式“搜一下/查一下”触发。
每群每天最多 20 次。
每用户每天最多 5 次。
```

## 聊天记录总结

短期总结用于回答：

- “刚才聊了什么”
- “总结一下这段聊天”
- “帮我整理一下刚才的结论”

输入范围：

- 最近 N 条消息。
- 默认不超过 100 条。
- 不包含非目标群消息。

输出要求：

- 用要点列出。
- 区分事实、结论和待确认事项。
- 不暴露个人隐私。
- 不做敏感属性推断。

长期摘要应在 SQLite 持久化后再做。它不是逐字保存无限聊天，而是压缩成群级摘要：

- 最近讨论主题。
- 常见问题。
- 未完成事项。
- 群内偏好的回复风格。

长期摘要必须可清理：

```text
/bot memory clear
/bot summary clear
```

## 群友特征记忆

允许记录的特征：

- 常聊主题，例如 Python、Docker、游戏、考试。
- 回复偏好，例如喜欢短回答、喜欢步骤清单。
- 明确自述的非敏感偏好，例如“我喜欢用 Windows”。
- 与机器人交互中的稳定偏好，例如“经常让机器人总结”。

禁止记录或推断：

- 政治倾向。
- 宗教信仰。
- 健康状况。
- 经济状况。
- 情感隐私。
- 年龄、性别、身份等敏感属性推断。
- 争吵中的负面人格标签。
- 未经确认的私人关系。

实现原则：

- 先做群级摘要，再做用户级轻量偏好。
- 用户级记忆必须可查看、可删除。
- 记忆内容要带来源类型，例如“用户自述”或“互动偏好”，不要伪装成事实。
- 不把特征记忆用于嘲讽或攻击。

## 安全策略

安全策略要分级，避免“一次拒答后所有问题都拒答”。

### 分级

1. 正常问题：直接回答。
2. 粗口、吐槽、轻微玩笑：可以轻描淡写接住，不必拒答。
3. 争吵：不站队，不拱火，把讨论拉回具体事实。
4. 隐私：不曝光、不总结、不推断个人隐私。
5. 违法或高风险指导：拒绝，并提供安全替代。
6. 敏感但可正常讨论：中性、简短回答，不无限回避。

安全策略应输出“替代路径”，而不是只拒绝。

## 图片能力

图片能力分两类。

### 读图

适合场景：

- 用户发截图让机器人解释。
- 用户发错误截图让机器人排查。
- 用户发图片让机器人提取文字或描述内容。

要求：

- 需要协议端能拿到图片文件或 URL。
- 需要视觉模型支持图片输入。
- 大图需要限制大小。
- 图片内容可能包含隐私，默认不做长期存储。

优先级：中。

### 输出图片

适合场景：

- 生成简单配图。
- 生成表情包草图。
- 把总结转成图片卡片。

要求：

- 必须管理员可开关。
- 默认不自动生成。
- 用户明确要求“生成图片”才触发。
- 需要控制单日次数和单次成本。

优先级：低于读图。

## 语音能力

语音能力通过语音转文字实现：

```text
QQ 语音
  -> 协议端获取语音文件
  -> 转码为模型可接受格式
  -> 语音转文字
  -> 进入普通文本流程
```

风险和难点：

- OneBot 协议端对语音文件获取方式可能不同。
- 需要转码工具。
- 语音转文字有额外成本。
- 群语音可能包含隐私。

结论：

- 不进入 M3/M4。
- 作为 M6 可选能力。
- 如果协议端取语音文件不稳定，可以放弃。

## 模型和价格策略

当前默认：

```text
文本模型：doubao-seed-2.0-lite
接口形态：OpenAI-compatible chat/completions
```

后续能力按供应商能力选择，不强制同一家供应商：

- 文本：优先继续使用豆包当前模型。
- 搜索：优先评估火山方舟是否有稳定官方搜索工具；否则接独立搜索服务。
- 读图：优先评估火山方舟视觉模型；不满足再接其他视觉模型。
- 出图：单独评估图片生成服务，默认关闭。
- 语音：单独评估 ASR 服务，默认后置。

成本控制必须配置化：

```env
ENABLE_WEB_SEARCH=false
ENABLE_IMAGE_INPUT=false
ENABLE_IMAGE_GENERATION=false
ENABLE_VOICE_TRANSCRIPTION=false
DAILY_SEARCH_LIMIT_PER_GROUP=20
DAILY_SEARCH_LIMIT_PER_USER=5
DAILY_IMAGE_LIMIT_PER_GROUP=5
BOT_MAX_REPLY_CHARS=300
```

价格不能写死在代码中。每个供应商的价格需要在接入前从官方控制台或官方价格页确认，并记录到文档。

## 阶段规划

### M2.5：搜索、安全和成本策略

- 增加安全分级策略。
- 增加搜索工具接口，但默认关闭。
- 增加搜索触发策略。
- 增加工具调用日志和限频。
- 增加成本相关配置。

### M3：可控上线

- 管理员命令。
- 开关。
- 状态查看。
- 群限频和用户限频。
- SQLite 持久化。

### M4：聊天总结和记忆

- SQLite 消息存储。
- 最近聊天总结。
- 群级长期摘要。
- 用户级轻量偏好。
- 记忆查看和清理命令。

### M5：图片和多模态

- 读图。
- 图片输出。
- 图片能力限频和成本控制。

### M6：语音

- 评估协议端语音文件获取。
- 评估转码和 ASR。
- 如果成本或稳定性不合适，放弃。

### M7：多群配置

- 从单群扩展到多个群。
- 每个群独立配置开关、模式、人设、限频和高级能力开关。
- 不同群之间上下文、摘要和用户偏好隔离。
